## 📌 What We Built
Financial institutions lose billions of dollars every year to credit card fraud. While modern machine learning models are incredibly good at flagging suspicious activity, they usually act like "black boxes." When an AI blocks a card, human investigators are left digging through data to guess *why* the system made that choice. This slows down processing times and frustrates legitimate users.

Our team, **Hungry Ninjas**, built a solution that bridges this exact gap. We paired a fast, calibrated **XGBoost classification model** with an **Explainable AI (XAI)** framework using **SHAP**. 

Everything is wrapped up in an interactive **Streamlit dashboard** that allows fraud investigators to see high-risk threats instantly and view a visual breakdown of the top 3 factors driving the risk score for any single transaction.

## 📂 What Each File Does

Here is a straightforward look at what's happening under the hood of our repository:

### ⚙️ Core Scripts
* **`train.py` (The Machine Learning Pipeline):** The engine of our project. It cleans the data, addresses the massive imbalance between fraud and normal transactions, trains our model, calibrates probabilities, and optimizes our decision thresholds.
* **`app.py` (The Dashboard Frontend):** Our interactive interface built with Streamlit. It reads the data and models generated by the backend to give investigators a clean, easy-to-use workspace with quick filters, metrics, and risk charts.

### 📊 Data & Inputs
* **`credit card fraud.csv` (Raw Dataset):** The baseline ledger of past credit card data containing details like amounts, transaction hours, location flags, and known fraud historical tags (`is_fraud`).
* **`scored_transactions.csv` (The Scored Output):** Generated by our training script. It saves the test data and automatically adds three intelligent columns: the 0-100 `risk_score`, the final `predicted_fraud` choice, and a text snippet of the `top_reasons` for easy reading.

### 🧠 Saved Model Brains & Encoders
* **`fraud_model_base.joblib`:** The raw, trained XGBoost model weights. The frontend dashboard feeds this file straight into our SHAP explainer to generate localized charts on demand.
* **`fraud_model.joblib`:** The calibrated model. It ensures the mathematical scores spit out by the AI accurately represent true real-world percentages (e.g., an 85% risk score actually means an 85% probability of fraud).
* **`merchant_category_encoder.joblib`:** A data translator. It turns plain text categories (like "Gas Station" or "Online Retail") into numbers the model understands.

### 📝 Configurations & Reports
* **`feature_cols.json` / `threshold.json`:** Small helper configuration files that tell the dashboard exactly which columns to look at and what mathematical cutoff line to use when separating normal charges from fraud.
* **`requirements.txt`:** The cloud checklist that tells Streamlit which Python libraries to install in the background so the app runs smoothly.
* **`RESULTS.md`:** A quick performance report card created during training to log precision, recall, and processing speeds.

## 🚀 Key Engineering Highlights

### 1. Zero Data Leakage (Chronological Splitting)
Instead of using a random split which allows a model to accidentally peek into future transaction trends, we sorted the data by time using `transaction_id`. We trained on the first 70% and tested on the final 30%. This mirrors how a real bank operates.

### 2. High Precision & Optimized Cutoffs
A standard 50% guessing line doesn't work for fraud detection. We evaluated the entire Precision-Recall Curve during training to pick an optimized threshold that keeps our precision high while catching the absolute maximum amount of fraud possible.

### 3. Built for Production Speeds
In real banking, you cannot wait seconds for a fraud check. Our complete inference pipeline clocks in at **~1.2ms** per transaction—well within strict production engineering budgets (typically under 200ms).
